
Jasne, Radosław — poniżej masz wersję zoptymalizowaną pod GitHub Copilot (GPT‑5.4 / GPT‑5.3‑codex) dla wszystkich 10 grup, w 3 trybach:
• Educational — Copilot tylko komentuje i tłumaczy
• Ultra‑Safe — Copilot wykonuje tylko 100% pewne zmiany
• Aggressive — Copilot robi maksymalnie dużo, ale bez zmiany logiki

Każdy prompt jest:
◇ krótki,
◇ imperatywny,
◇ checklistowy,
◇ zrozumiały dla Copilota,
◇ działający na module/pakiecie,
◇ bez długich opisów, które Copilot ignoruje.

To jest wersja finalna, produkcyjna, gotowa do użycia w IDE.

⭐ 1. NAMING — Copilot‑Optimized Prompts

🎓 Educational
Do NOT modify code.  
Review this module/package and comment on naming issues.  
Identify unclear names, abbreviations, generic method names, and boolean naming problems.  
Explain why they are problematic and propose better technical names.

🛡 Ultra‑Safe
Refactor naming in this module/package using only safe technical changes:  
- expand abbreviations  
- improve boolean names  
- replace generic method names with technical ones  
- improve variable names based on usage  
Do NOT rename domain concepts or public APIs.  
Apply changes only inside this module.

🔥 Aggressive
Aggressively refactor naming in this module/package:  
- expand all abbreviations  
- rename generic methods  
- unify naming conventions  
- rename helper/manager/utils classes  
- improve all variable names  
Do NOT change business logic.  
Apply changes only inside this module.

⭐ 2. LOGIC SIMPLIFICATION

🎓 Educational
Do NOT modify code.  
Review this module/package and comment where logic can be simplified:  
guard clauses, boolean extraction, negation removal, boolean flags.  
Explain improvements with examples.

🛡 Ultra‑Safe
Simplify logic safely in this module/package:  
- convert nested ifs to guard clauses  
- extract complex boolean expressions  
- remove unnecessary negations  
- simplify boolean comparisons  
Do NOT change logic or method signatures.

🔥 Aggressive
Aggressively simplify logic in this module/package:  
- rewrite nested conditionals  
- extract all multi‑condition expressions  
- remove boolean flags (split methods if needed)  
- unify logical structure  
Do NOT change business logic.

⭐ 4. NULL SAFETY

🎓 Educational
Do NOT modify code.  
Review null‑safety issues: missing annotations, null returns, missing validation, Optional usage.  
Explain improvements.

🛡 Ultra‑Safe
Improve null‑safety safely:  
- add @NotNull/@Nullable based on usage  
- replace return null with empty collections  
- add Objects.requireNonNull at boundaries  
Do NOT introduce Optional unless already used.

🔥 Aggressive
Aggressively improve null‑safety:  
- annotate all parameters and returns  
- replace all null returns with safe alternatives  
- convert null‑handling to Optional where appropriate  
- enforce fail‑fast validation

⭐ 5. IMMUTABILITY

🎓 Educational

Explain improvements.

🛡 Ultra‑Safe

🔥 Aggressive
Aggressively enforce immutability:  
- add final everywhere possible

- replace wrappers with primitives  
- mark fields final when safe
